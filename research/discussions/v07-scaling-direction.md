# V07 Discussion: 64-Node Results Analysis and V8 Direction

**Date**: 2026-03-08
**Context**: V7 64-node experiment shows fault impact diluted beyond detection, three experts invited to analyze.

---

## Panel 1: 排隊論專家

### M/D/1 模型分析

每個 worker 是 M/D/1 佇列（Poisson arrival, deterministic service 10ms, single server）。

**正常節點平均響應時間**：
```
W_q = ρ·S / 2(1-ρ) = 0.80 × 10 / (2 × 0.20) = 20ms
T = W_q + S = 30ms  （與實驗 ~29ms 吻合）
```

### 1/64 故障不可見的數學解釋

整體延遲分佈為混合分佈：`F(t) = (63/64)·F_normal(t) + (1/64)·F_faulty(t)`

P99 要求 F(t)=0.99：即使故障節點所有請求都無限慢，正常節點的 P99 就已覆蓋 99%。
1/64 = 1.56% 的壞請求只把 P99 往上推不到 1ms。

### P2C 自然迴避的定量估計

慢節點被 P2C 選中的機率 ≈ (f/N)² = (1/64)² = 0.024%（當佇列差異明顯時）。
10x slowdown 節點的佇列長度約正常的 10 倍，P2C 幾乎不會選中它。

**局限**：在 2-3x 輕微 slowdown 時，佇列差異小，P2C 區分力顯著下降 → **這是 mitigation 的價值區間**。

### 故障節點臨界數量（64 nodes, 80% load）

| 故障節點 f | 正常節點 ρ | M/D/1 平均響應 | 近似 P99 |
|-----------|-----------|--------------|---------|
| 0 | 0.800 | 30.0ms | ~45ms |
| 4 | 0.853 | 39.1ms | ~62ms |
| 6 | 0.883 | 47.8ms | ~78ms |
| 8 | 0.914 | 63.2ms | ~105ms |
| 10 | 0.949 | 107.8ms | ~190ms |
| 12 | 0.985 | 340ms | 爆炸 |

**P99 > 50ms 需要 ~4 nodes, > 100ms 需要 ~8 nodes**（已由 S4 數據驗證）。

### 建議的 V8 參數

- 基礎 load_factor: **0.88-0.90**（膝蓋區域，每 1% ρ 變化 = ~5ms 延遲變化）
- Slowdown: **2-3x**（P2C 盲區，更 realistic）
- 故障節點: S1 改 4/64, S4 改 8/64, S6 改 4→8 級聯

## Panel 2: 分散式系統專家

### 生產環境故障比例

- AWS/Google/Meta：任意時刻約 **0.1%-2%** 節點性能退化
- Gray failure（灰色故障）佔生產事故顯著比例，比完全宕機更頻繁
- "The Tail at Scale" (Dean & Barroso, 2013)：fan-out 下 1% 慢請求影響 63% 使用者

### 架構模型的根本問題

當前模型是 **單層、無依賴、可完全替代** 的架構：
```
Client → Load Balancer → [64 homogeneous workers]
```

生產環境缺少的關鍵因素：
| 因素 | 當前模型 | 生產環境 |
|------|---------|---------|
| 依賴深度 | 1 層 | 3-7 層 |
| Fan-out | 1:1 | 1:N（scatter-gather） |
| 資源共享 | 無 | CPU/network/disk 競爭 |
| 故障傳播 | 不傳播 | 背壓、連鎖超時 |
| 節點可替代性 | 完全可替代 | 有狀態、shard affinity |

**結論**：64 nodes 本身沒問題，但缺乏架構複雜度。

### Fan-out 模型的影響

| Fan-out | 碰到慢節點機率 (1/64 故障) | 碰到慢節點機率 (4/64 故障) |
|---------|---------------------------|---------------------------|
| 1 | 1.56% | 6.25% |
| 5 | 7.58% | 27.6% |
| 10 | 14.6% | 47.8% |
| 20 | 27.0% | 73.4% |

Fan-out=10 + 4 故障節點 → 接近一半請求被影響。這是最能改變實驗結論的架構改動。

### 策略在不同負載區間的價值

```
              低負載(<50%)  中負載(50-75%)  高負載(75-90%)  過載(>90%)
SPECULATE     有效 ✓        有效 ✓          危險 ⚠          有害 ✗
SHED          不需要         不需要          有效 ✓          關鍵 ✓✓
ISOLATE       有效 ✓        有效 ✓          有效 ✓          有效 ✓
```

### 建議的改進方向（按優先級）

| 優先級 | 改動 | 預期收益 | 工作量 |
|--------|------|---------|--------|
| P0 | 調參數：高故障比例 + 高負載 | 立即看到差異 | 小 |
| P0 | 增加指標：P999、goodput、TTD | 即使延遲差異小也能區分 | 小 |
| P1 | **Fan-out 模型** | 根本性改變故障影響面 | 中 |
| P2 | 多層依賴鏈 | 展示策略適用場景 | 大 |
| P3 | 共享資源競爭模型 | 展示 SHED 獨特價值 | 大 |

### 額外場景建議

- **S-A 高故障比例**：16/64 nodes 故障（模擬 AZ 級降級），80% load → rho_eff = 106%
- **S-B 極高負載**：2/64 故障，95% load → 幾乎沒餘量
- **S-C 漸進蔓延**：每 30 秒增加 2 故障節點，最終 20/64，85% load

## Panel 3: 實驗設計專家

### Sweet Spot 公式

```
rho_eff = rho_base / (1 - f × (1 - 1/s))
```

- rho_eff < 0.85：系統自癒，策略無用（V7 的 S1/S2/S3/S5/S7）
- rho_eff > 0.95：系統飽和，策略有害（V6 的 S3/S4）
- **Sweet spot: 0.85 < rho_eff < 0.95**（V6 的 S6，V7 的 S4/S6）

### 叢集大小建議：32 nodes

| 大小 | 優勢 | 劣勢 |
|------|------|------|
| 16 | 故障影響大 | 單節點 = 災難級，不 realistic |
| **32** | **折衷**：1n=3.125%, 4n=12.5% | — |
| 64 | 更 realistic | 單節點故障不可見 |

### 必須新增的指標

| 指標 | 定義 | 為什麼重要 |
|------|------|-----------|
| P999 latency | 99.9th percentile | P99 被稀釋時 P999 仍敏感 |
| Tail ratio | P99/P50 | 正規化，消除基礎延遲差異。健康 ~2-3x，故障 >10x |
| SLO violation rate | count(lat > SLO) / total | 直接衡量 SLO 合規 |
| Windowed max VR | 每 5s 窗口最大 violation rate | 捕捉暫態惡化 |
| Goodput | SLO 內完成的 req/s | 比 throughput 更真實 |
| Time-to-detect | 故障→首次觸發 mitigation | 偵測器效能 |
| Recovery time | 恢復→P99 回基線 ±10% | S7 核心指標 |
| Affected ratio | 被路由到故障節點的比例 | 驗證 LB 重分配效果 |
| Mitigation cost | hedge_count + shed_count | 策略代價 |

### 完整 V8 實驗方案

**全域參數**：
```
N_WORKERS = 32
SLO_TARGET = 50ms（降低，貼合基礎延遲）
DURATION = 120s
N_RUNS = 20（提高統計信心）
```

**場景設計**：

| 場景 | 故障節點 | Slowdown | Load | rho_eff (peak) |
|------|---------|----------|------|----------------|
| S1 sweep | 1/32 | 1x-20x | 0.80 | 0.80-0.83 |
| S2 progressive | 2/32 | 1→15x, β=0.1 | 0.80 | →0.84 |
| S3 flash crowd | 2/32 | 3x | 0.70→0.85→0.90 | 0.89→0.94 |
| S4 multi-node | 4/32 | 2-5x | 0.80 | 0.88 |
| S5 fluctuating | 3/32 | 5x on/off | 0.80 | 0.80↔0.87 |
| S6 cascade | 1→2→3/32 | 5x/5x/3x | 0.80 | 0.81→0.87 |
| S7 recovery | 2/32 | 8x | 0.80 | 0.84→0.80 |

**驗證流程**：先跑 S1 確認 sensitivity curve，確認 P999/tail ratio 能區分後再跑全部。

## 三方共識

1. V7 故障影響比太低，不是策略問題
2. P2C 在大叢集下自然迴避慢節點，讓 1/64 故障不可見
3. 需要 P999、tail ratio、goodput 等多維指標
4. 降低 slowdown 到 2-3x（P2C 盲區）更 realistic
5. 提高負載到膝蓋區域（0.85-0.90）讓區分度最大化

## 三方分歧

| 議題 | 排隊論 | 分散式系統 | 實驗設計 |
|------|--------|-----------|---------|
| 叢集大小 | 64n, 調故障比 | 64n, 加 fan-out | **32n** |
| 架構改動 | 不需要 | **Fan-out（P1）** | 不需要 |
| SLO target | 維持 100ms | 不置可否 | **降到 50ms** |

## 建議路線

P0（立即可做）：調參數 + 加指標 → 驗證 sensitivity
P1（中期）：Fan-out 模型 → 根本性改變故障影響面
P2（長期）：多層依賴鏈 + 共享資源
